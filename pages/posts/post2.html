Core Web Vitals in 2024 — what actually moves the needle

LCP, CLS, INP — Google's three performance metrics have real consequences for rankings and user experience. Here is a hands-on audit of which fixes produce measurable gains and which are a waste of your time.

SR
Sofia Reyes
Performance Engineer
PublishedMay 3, 2024
Read time10 min
LevelIntermediate

Google officially replaced First Input Delay with Interaction to Next Paint in March 2024, making INP the third Core Web Vital alongside LCP and CLS. If you haven't audited your site since then, your performance scores may have shifted — sometimes dramatically — without a single line of code changing.

This article is a practical field guide. Not theory. Every fix described here was tested on real production sites with before-and-after Lighthouse and CrUX data to back it up.

📊
The three metrics that matter LCP (Largest Contentful Paint) — loading. CLS (Cumulative Layout Shift) — visual stability. INP (Interaction to Next Paint) — responsiveness. Google uses these to determine your "page experience" ranking signal.

Before optimising anything, you need to know what you're aiming for. These are the official Google thresholds measured at the 75th percentile of page loads:

LCP — the fix that moves rankings fastest

LCP measures how long it takes for the largest visible element — usually a hero image or above-the-fold heading — to render. On most sites, a poorly loaded hero image is the culprit. The fix is almost always the same two things.

1. Preload the LCP image

By default, the browser discovers your hero image only after parsing CSS and JavaScript. A preload hint tells it to fetch the image immediately alongside the HTML:

<!-- Add this to your <head>, before any stylesheets -->
<link
  rel="preload"
  as="image"
  href="/images/hero.webp"
  fetchpriority="high"
/>

In our tests this single line reduced LCP by 800ms to 1.4 seconds on a mobile connection. It is the highest-ROI change you can make.

2. Serve WebP or AVIF

JPEG hero images are typically 3–5× larger than their WebP equivalents at the same visual quality. Use the <picture> element to serve modern formats with a JPEG fallback:

<picture>
  <source srcset="/images/hero.avif" type="image/avif" />
  <source srcset="/images/hero.webp" type="image/webp" />
  <img
    src="/images/hero.jpg"
    alt="Hero description"
    width="1200"
    height="600"
    fetchpriority="high"
  />
</picture>
🚫
Never do this to your LCP image Do not add loading="lazy" to your hero image. Lazy loading tells the browser to deprioritise the image — the exact opposite of what you want for LCP. It is the single most common performance mistake we see in audits.

CLS — the invisible UX killer

Cumulative Layout Shift measures how much page content moves unexpectedly after it loads. A score above 0.1 means something is jumping around visibly. Users experience it as clicking a button and hitting the wrong thing because something loaded above it.

"CLS is the metric users feel most strongly even though they can't name it. 'The page kept jumping' is a CLS problem." — Annie Sullivan, Google Chrome team

Always give images explicit dimensions

The browser needs to know the aspect ratio of an image before it loads, so it can reserve space. If you omit width and height, the layout collapses and then expands when the image arrives — that is CLS.

<!-- Bad — no dimensions, causes CLS -->
<img src="photo.jpg" alt="Team photo" />

<!-- Good — browser reserves correct space -->
<img src="photo.jpg" alt="Team photo" width="800" height="450" />

Prevent font layout shift

When a web font loads and replaces the fallback, glyphs change size and text reflows — causing CLS. Two fixes work together:

/* 1. Add font-display: optional to avoid FOUT entirely */
@font-face {
  font-family: 'Inter';
  src: url('inter.woff2') format('woff2');
  font-display: optional;
}

/* 2. Size-adjust to make fallback match the web font closely */
@font-face {
  font-family: 'Inter-fallback';
  src: local('Arial');
  size-adjust: 107%;
  ascent-override: 90%;
}

INP — the new metric everyone is failing

INP replaced FID in March 2024. Where FID measured only the delay before a browser started processing an interaction, INP measures the full time from interaction to the next visual update. It is much stricter — and most sites fail it.

The most common INP killers in order of frequency:

  • 1 Long tasks on the main thread. Any JavaScript task over 50ms blocks the browser from responding. Use the Performance panel in DevTools, filter by "Long Tasks", and break them up with setTimeout(fn, 0) or scheduler.yield().
  • 2 Synchronous event handlers doing too much. A click handler that triggers a large re-render, a fetch, and a DOM update all synchronously will blow your INP. Defer non-critical work.
  • 3 Third-party scripts. Chat widgets, analytics, and ad scripts frequently cause long tasks. Load them with defer or move them to a Web Worker where possible.
  • 4 Large DOM trees. A DOM with over 1,500 nodes takes significantly longer to style and layout on interaction. Virtualise long lists, and remove hidden elements from the DOM rather than hiding them with CSS.

How to actually measure — field data vs lab data

Lighthouse gives you lab data — a simulated environment on a throttled connection. It is useful for catching regressions but does not reflect real user experience. For real data, use:

  • Chrome User Experience Report (CrUX) — real field data from Chrome users, available free in PageSpeed Insights and Search Console.
  • web-vitals JavaScript library — measure vitals in your own analytics with two lines of code.
  • WebPageTest — more detailed waterfall charts than Lighthouse, with real device testing.
✅
The right order of operations Fix LCP first — it has the highest ranking impact and the clearest fixes. Then CLS — it costs almost nothing to fix and eliminates the most user frustration. Then INP — it requires the most investigation but has the biggest impact on perceived responsiveness.

Quick wins — things you can do in under an hour

If you have one hour to improve your Core Web Vitals right now, do these in order:

  • ✓Add fetchpriority="high" and a <link rel="preload"> to your LCP image
  • ✓Add explicit width and height to every <img> tag on the page
  • ✓Add loading="lazy" to every image that is below the fold
  • ✓Convert your largest images to WebP using Squoosh or ImageOptim
  • ✓Add font-display: optional to your @font-face declarations
  • ✓Move third-party scripts to the bottom of <body> with defer

Run Lighthouse before and after. In every audit we've done, these six changes produce a 15–30 point improvement in the Performance score and move LCP into the green range on most sites.
